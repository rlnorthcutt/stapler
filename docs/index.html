The Problem

Browsers weren't built for staples.

"Web browsers were built for infinite scrolling. Business reports were built for staples. Between the two lies a graveyard of broken CSS @media print rules and overlapping footers."

The Mechanism

Three reasons it actually works.

Hard-Clipped Precision

We don't guess. If it doesn't fit on the page, it's gone — cleanly, intentionally, with overflow: hidden instead of a prayer. Build documents like an engineer, not a typesetter.

AI-Native

LLMs are exceptional at writing structured content. They are terrible at calculating millimeter margins. Stapler is designed so an AI can emit valid, predictable markup without understanding CSS layout.

Zero Drama

No NPM. No build step. No 2 GB node_modules folder having an existential crisis in your project root. One <script> tag. Six custom elements. Done.

5-Second Setup

From zero to page one.

Drop in the script tag and start writing pages. The library registers all six elements immediately.

Explicit Mode

You own the layout. Each <s-page> is a hard-clipped container — position anything inside it absolutely, build two-column spreads, or leave a page blank as a separator.

<stapled-doc mode="explicit"
  page-width="8.5in"
  page-height="11in"
  page-gap="2rem">

  <page-header height="36px" skip-first>
    My Report · <page-number format="n of total"></page-number>
  </page-header>

  <s-page skip-header skip-footer>
    <!-- Cover page -->
  </s-page>

  <s-page>
    <!-- Page 2 -->
  </s-page>

</stapled-doc>

Flow Mode

Write content naturally. The library measures, computes page boundaries, and injects invisible spacers to push content out of header and footer zones — automatically.

<stapled-doc mode="flow"
  page-width="8.5in"
  page-height="11in"
  page-gap="2rem">

  <page-header height="36px">
    <page-number format="n of total"></page-number>
  </page-header>

  <h1>Executive Summary</h1>
  <p>Content flows here...</p>

  <page-spacer></page-spacer>

  <h2>Chapter Two</h2>

</stapled-doc>
